Section 1: Competitive Teardown — How the Big
Platforms Actually Work
To build a discovery engine that structurally outcompetes the incumbent streaming monopolies, we
must first deconstruct their recommendation architectures at the tensor level. The music streaming
industry has converged on a set of machine learning paradigms that are highly optimized for user
retention, session length, and platform monetization [1]. However, these exact architectural
choices create an insurmountable structural barrier for independent artists and niche genre
discovery. This section provides a deeply technical teardown of the recommendation pipelines
powering Spotify, Apple Music, YouTube Music, and SoundCloud. By mapping their retrieval,
ranking, and re-ranking layers, we expose the mathematical realities of their popularity biases and
define the exact architectural voids that the Harmony Discovery Engine will exploit.
1.1 Spotify: The Gradient Starvation Machine
Spotify operates the most sophisticated machine learning ecosystem in the audio streaming
market, utilizing dozens of specialized AI agents and ML models operating across multiple levels
[1]. Their architecture has recently transitioned from traditional embedding-based retrieval to
generative retrieval and graph-based foundation models [2][3]. Spotify's recommendation
infrastructure follows a massive funnel-style architecture, reducing a catalog of over 100 million
tracks to a personalized UI in milliseconds.
Three-Stage Pipeline Architecture
Stage 1 — Retrieval: Spotify historically relied on a Two-Tower (2T) model to place user and item
vectors in a shared embedding space, using Approximate Nearest Neighbor (ANN) search for
candidate generation [3][4]. Recently, they have pioneered Generative Retrieval via projects like
Text2Tracks and GLIDE [2][5]. Instead of embedding lookups, they fine-tune an LLM to directly
generate optimized track IDs from user prompts or contexts [2]. This is achieved by mapping
tracks to Semantic IDs — discrete tokens created via vector discretization over song

characteristics and collaborative-filtering embeddings [2]. The LLM acts as a catalog-native reasoning engine, generating both
natural language and Semantic IDs simultaneously [6].
Stage 2 — Ranking: The ranking layer is dominated by the 2T-HGNN (Two-Tower Heterogeneous
Graph Neural Network) [4]. The HGNN operates on a massive co-listening graph to produce
structural item-to-item and user-to-item signals [3]. This is combined with Generalized User
Representations (GURs) introduced in 2025, which frame dynamic taste profiles across
consumption contexts based on the co-occurrence of songs in playlists and listening sessions [7].
For sequential modeling, BaRT (a BERT-style transformer) evaluates the user's immediate session
history to predict the next optimal track, heavily weighting recent skips and completions.
Stage 3 — Re-Ranking: The final layer applies business logic and exploration algorithms. Spotify
utilizes ICEE (In-context Exploration-Exploitation), which performs Bayesian belief updates via
neural network inference without requiring gradient optimization, allowing real-time adaptation to
user preferences within tens of episodes [8]. This layer also injects promoted content (Discovery
Mode) and enforces diversity constraints.
Signals Spotify Uses
• Implicit Signals: Stream length, skip timing, repeat plays, playlist adds, library saves, share
events, search-then-play behavior, and listening session context [1]
• Explicit Signals: Thumbs up/down, explicit ratings, and user-generated playlist curation [7]
• Contextual Signals: Time of day, device type, and inferred user environment or state [9]
• Social & Graph Signals: Follower graphs, collaborative playlist co-occurrence, and the
massive co-listening graph powering the HGNN [3]
• Audio Features: BPM, key, energy, valence, rhythm, timbre, and production style, often
extracted via CNNs operating on audio spectrograms [10][9]
• NLP & Metadata: Playlist title embeddings, artist bio text, lyrics, and blog sentiment [7][1]
The Structural Popularity Bias Flaw — Why Spotify Cannot Fix It
Despite its sophistication, Spotify's architecture contains a fatal, unpatchable flaw regarding
independent artist discovery. The system is mathematically bound to a popularity bias that starves
niche content of gradient updates.
Gradient Starvation in Interaction Logs: Spotify's models are trained primarily on user interaction
logs. In machine learning, the magnitude and frequency of gradient updates dictate a model's
understanding of an entity. A track with 10,000,000 streams generates roughly 10,000x more
gradient signal than a track with 1,000 streams. The neural networks literally lack the mathematical
vocabulary to accurately represent the niche track because it rarely appears in the training batches.
Co-occurrence Dependency in 2T-HGNN: The Heterogeneous Graph Neural Network relies on a
co-listening graph [3]. Edges in this graph are formed when users listen to two tracks in proximity.
If a track is brand new or highly niche, it has a node degree approaching zero. Tracks that never
appear together cannot develop meaningful similarity relationships in the graph space. The HGNN

structurally ignores disconnected nodes.
BaRT Sequence Invisibility: BaRT and other sequential models are trained on session sequences.
Niche tracks appear in so few user sessions that they are effectively invisible to the sequence
model's attention mechanism. The model learns to predict high-probability transitions (mainstream
track A rightarrow mainstream track B) and penalizes low-probability transitions.
Semantic ID Quantization Collapse: Spotify's GLIDE and Text2Tracks rely on Semantic IDs, which
are generated via vector discretization (quantization) over collaborative-filtering embeddings [2][5].
The VQ-VAE codebook used to create these discrete tokens is dominated by popular track
patterns. When a highly unique, niche track is quantized, it is often forced into a generic 'noise'
bucket or miscategorized because the codebook lacks the resolution to represent rare sonic
profiles accurately.
Business Model Lock-in: Spotify cannot flatten this popularity gradient because their business
model depends on it. Major label deals, the 'Discovery Mode' promoted content revenue, and artist
marketing programs all rely on the predictable amplification of popular content. Flattening the
gradient to favor true audio-based discovery would destroy label relationships and cannibalize their
promoted insertion revenue. They structurally CANNOT fix this.
SPOTIFY CANNOT DO THIS: Spotify cannot treat a brand-new track from an unknown artist with
the same recommendation confidence as a track with 10M streams — their entire training pipeline
is built on interaction volume as a proxy for quality.

recommendation. Rather than relying purely on deep learning and interaction graphs, Apple
employs a hybrid system that heavily weights human editorial curation alongside algorithmic
amplification. The top of the funnel is controlled by human editors who curate flagship playlists
(e.g., New Music Daily, Today's Hits, A-List by Genre). The algorithmic layer — powering surfaces
like the New Music Mix, Personal Mix, and Now Playing radio — is built on top of these editorial
signals and collaborative filtering derived from the Apple ecosystem [11].

1.3 YouTube Music: Watch-Time Optimization Suppresses Niche
Discovery
YouTube Music inherits the massive, battle-tested recommendation infrastructure of YouTube's
core video platform, fine-tuned for audio and music video consumption. YouTube utilizes a
two-stage Deep Neural Network (DNN) recommendation system, originally codified in their
landmark 2016 paper and continuously updated [12][13][14][15].
Candidate Generation Network: This network reduces a corpus of millions of videos/tracks down to

a few hundred highly relevant candidates [12][16]. It frames recommendation as extreme multiclass classification. User activity history
(watch IDs, search queries, demographics) is embedded and passed through a feed-forward network to predict a softmax distribution
over the entire catalog [12][14][15]. It relies on approximate nearest neighbor search for millisecond-level inference [15].
Ranking Network: The ranking stage uses a separate deep neural network to assign an impression
score to each candidate [14]. It utilizes a rich, multi-modal feature set (hundreds of features) and
weighted logistic regression [15]. Crucially, positive interactions are weighted by watch time to
optimize for overall engagement rather than simple click-through rates [15]. YouTube's system
continuously learns from over 80 billion signals [19].
The Structural Flaw — Watch Time Optimization Trap: YouTube's structural flaw lies in its core
optimization metric: expected watch time [14][16][15]. The architecture is explicitly designed to
maximize the time a user spends on the platform. This creates a feedback loop that heavily favors
familiar, high-production, mainstream content over genuinely novel niche discovery. In the
candidate generation phase, the extreme multiclass classification uses negative sampling [15].
Items with low prior probabilities (niche tracks) are heavily penalized. A user might discover a
brilliant niche track, listen to it once, and feel satisfied. Conversely, they might leave a mainstream
pop playlist running in the background for hours. YouTube's ranking network will overwhelmingly
favor the background pop music because it yields higher total watch time, actively suppressing the
high-satisfaction niche discovery.
THE HARMONY ADVANTAGE: Harmony optimizes for discovery satisfaction (e.g., did the user
explore a new artist and save their track?) rather than raw session length. By utilizing ALS
collaborative filtering combined with Essentia audio features, Harmony builds cross-genre bridges
based on sonic similarity that watch-time optimization would actively suppress.
1.4 SoundCloud: Social Capital Dependency
SoundCloud has historically been the haven for independent artists and underground electronic
music, but its recommendation architecture is fundamentally constrained by its reliance on social
mechanics. SoundCloud's core discovery mechanism is the repost graph. The social propagation
of tracks through user reposts dictates visibility. While they utilize collaborative filtering for
'Suggested Tracks' based on repost and like behavior, the social graph is the engine.
To address the cold-start problem (the 'Zero Plays problem'), SoundCloud recently introduced the
'First Fans' initiative, powered by AI technology from their acquisition of Musiio [20][21]. This
system uses content-based filtering — analyzing recording quality, mood, genre, and energy — to
recommend newly uploaded tracks to approximately 100 real users whose listening history aligns
with the track's style [20][21]. The engine processes thousands of candidate matches per second
using real-time data pipelines [20].
Despite the introduction of 'First Fans,' SoundCloud's architecture is still fundamentally a social
network. The repost graph creates popularity cascades — tracks either go viral through
interconnected repost chains or die in complete obscurity. An artist with no established follower
base has no initial node to begin a repost propagation. While 'First Fans' attempts to bypass this

1.5 The Five Universal Failure Patterns
When analyzing Spotify, Apple Music, YouTube Music, and SoundCloud at the architectural level,
five distinct failure modes emerge. These are not bugs; they are the intended mathematical
outcomes of their system designs.
1. Popularity Bias (Gradient Starvation): All four platforms use interaction volume as their
primary training signal proxy. Whether it is Spotify's co-listening HGNN [3], YouTube's
watch-time weighted logistic regression [15], or Apple's reliance on Shazam volume [11], the
models are mathematically biased toward tracks with massive historical data. The rich get
richer by design.
2. Cold Start Blindness: None of the incumbents possess a robust, purely audio-first cold start
mechanism that scales. Spotify requires session co-occurrence [7], Apple requires editorial
selection or early compounding [11], and SoundCloud's 'First Fans' is a limited 100-user
band-aid over a social graph [20].
3. Metric Misalignment: YouTube and Spotify optimize for engagement, session length, and
retention [1][16]. This metric misalignment prioritizes passive, familiar background listening
over active, novel discovery, actively suppressing niche genres that require attentive listening.
4. Metadata Dependency: Platforms rely heavily on NLP of lyrics, blog sentiment, and
user-supplied tags [7][1]. For underground electronic, experimental bass, or instrumental
niche genres, this metadata is either non-existent or highly inaccurate, rendering text-based
retrieval useless.
5. Business Model Conflicts: The most critical failure point. Spotify's Discovery Mode, Apple's
major label relationships, and YouTube's ad-inventory requirements mean that even if their
engineers wanted to flatten the popularity curve to favor independent artists, their executive
boards would veto it. The bias is monetized.

Section 2: The Harmony Algorithm Architecture
This is the moat. Every streaming giant has collaborative filtering. Every music app has genre tags.
What none of them have optimized for is the independent artist discovery problem — where 80%
of the catalog has fewer than 1,000 streams and cold start is the default state, not the exception.

2.1 System Overview
The Harmony recommendation engine is a 4-stage pipeline: raw audio and behavioral signals flow
through ingestion, get transformed into multi-modal candidate sets, get ranked by a learned
scoring function, and get continuously refined by feedback. Every stage is designed to be
independently replaceable as the system scales.
Why 4 layers? Each layer captures a fundamentally different signal about musical affinity. Audio
similarity catches sonic texture. Collaborative filtering catches 'people like you loved this.' Genre
graph catches structural musical relationships. Cold start ensures new tracks aren't invisible. No
single layer is sufficient — the magic is in the ensemble. Candidate budget: Each layer generates
~100–200 candidates. After deduplication, ~400–500 unique tracks enter the ranking stage. The
ranker then produces a final ordered list of 50 recommendations, with 5 serendipity slots injected.

a festival closer.' But CLAP is a black box. You can't interrogate why two tracks are similar.
Essentia gives you the forensics: exact BPM, MFCC coefficients, spectral centroid, key/scale.
These features are interpretable, filterable, and composable. You can say 'find me tracks within

2.3 Layer B: Collaborative Filtering (ALS with Popularity Penalty)
Collaborative filtering answers the question: 'What do people with similar taste to this user love?' It
doesn't care about audio at all — it's pure behavioral signal. A user who loves Burial, Andy Stott,
and Actress will get recommended Demdike Stare not because it sounds similar (though it does)
but because the same humans who love those artists also love Demdike Stare. For independent
artists, this is both the most powerful signal and the most dangerous one. Powerful because it
captures taste in ways audio analysis never can. Dangerous because it naturally amplifies popular
tracks — the rich get richer. The solution: a popularity penalty that inverts the rich-get-richer
dynamic.

artists rather than just another platform that surfaces the same 10,000 tracks.
2.4 Layer C: Genre Graph (GraphSAGE GNN)
Music exists in a web of relationships that neither audio analysis nor collaborative filtering fully
captures. A track can be sonically similar to another but in a completely different genre context. A
user can love jazz and also love electronic music — two genres that rarely appear in the same
collaborative filtering neighborhood. The genre graph makes these structural relationships explicit
and learnable.
The heterogeneous graph structure contains four node types (Users, Tracks, Artists, Genres) and
six edge types (listened, followed, classified, tagged, sonically_similar, adjacent). Standard GNNs
(like GCN) are transductive — they can only generate embeddings for nodes that existed during
training. Every new track upload would require a full retraining. GraphSAGE is inductive. It learns
an aggregation function over neighborhoods, not embeddings for specific nodes. A new track can
get an embedding immediately by aggregating its neighbors' features — no retraining required.
Phase 1 Simplification: The full GraphSAGE implementation requires significant training
infrastructure. In Phase 1, replace it with a pre-computed genre similarity matrix. This gives you
80% of the value with 5% of the complexity. Compute genre-to-genre similarity from playlist
co-occurrence: count how often genres appear together in the same playlist, then normalize to
similarity scores (0–1) using PMI or cosine normalization.
2.5 Layer D: Cold Start (ACARec Artist Catalog Attention)
Cold start is not an edge case in Harmony — it's the default state. Every new track upload from an
independent artist starts with zero streams, zero interactions, zero collaborative filtering signal.
Without a dedicated cold start layer, new tracks are invisible. ACARec (Artist Catalog Attention
Recommendation) solves this by bootstrapping a new track's CF embedding from the artist's
existing catalog, weighted by audio similarity.

Section 3: The Niche Genre Discovery System
3.1 The Genre Taxonomy
The fundamental failure of modern music recommendation engines is their shallow understanding
of genre. Spotify, Apple Music, and Tidal operate almost exclusively at Level 1 (L1) or Level 2 (L2)
Built with Spine AI on 2026-06-23 16
genre specificity. To them, a track is 'Electronic' (L1) or 'House' (L2). For a casual listener, this is sufficient. But for the obsessive
music fan — the core demographic Harmony is built for — this shallow categorization is useless. A listener searching for the organic,
polyrhythmic textures of 'Afro-Organic Deep House' (L4) will be actively alienated if an algorithm serves them 'Big Room Progressive'
(L3) simply because both roll up to 'House' [27]. Operating at L4/L5 genre specificity is Harmony's core competitive advantage. No
platform has ever built a recommendation engine at this depth because the data engineering required to maintain a 6,000+ node
taxonomy and the machine learning required to classify audio into it are exceptionally difficult.
3.2 Taxonomy Build Strategy: Four-Source Aggregation
Building an L5 taxonomy from scratch is impossible. Instead, we aggregate, deduplicate, and
curate from the four most comprehensive genre databases on the internet, standardizing them into
a single unified hierarchy [27][28].
1. Every.Noise.at: The holy grail of genre data — a publicly accessible frontend for Spotify's
internal genre map containing over 6,000+ genres [27]. Parse the DOM, extract text nodes of
links to form the raw taxonomy, and map associated playlist URIs to extract reference tracks
for training [29][30].
2. MusicBrainz: An open-source, community-maintained database offering over 1,000+ genres
[27]. Access via their open API, paginating through
https://musicbrainz.org/ws/2/genre?limit=100&offset=0&fmt=json to pull the complete list
including community-vetted aliases and UUIDs [31].
3. Discogs: The definitive database for electronic and physical releases, containing over 15M+
releases [28]. Uses a two-tier system: Genre is top-level (e.g., Electronic, Rock), and Style is
the subgenre (e.g., Deep House, Techno). Extract via monthly XML data dumps, mapping
Style tags to L3/L4 nodes [28].
4. Rate Your Music (RYM): Possesses the most pedantic, highly debated community genre
taxonomy on the internet. Scrape genre hierarchy pages to understand parent-child
relationships between obscure micro-genres, critical for structuring L4 and L5 nodes.
The Merge and Deduplication Strategy: Ingesting these four sources yields roughly 12,000 raw
genre strings. Process through a deterministic normalization pipeline: convert to lowercase, strip all
punctuation and special characters, execute fuzzy string matching using Python rapidfuzz library
(Levenshtein distance), apply a similarity threshold of 85 (strings scoring above this threshold are
flagged as near-duplicates, e.g., 'tech-house', 'tech house', 'techhouse'), a manual curation pass
defines the canonical name, and all non-canonical variants are written to a genre_aliases table
ensuring incoming metadata from any source can be instantly mapped to canonical UUIDs.

3.4 CT-GateNet CNN-Transformer Genre Classifier
To operate at L4/L5 specificity, metadata alone is insufficient. User-generated tags are notoriously
noisy, and emerging artists often lack metadata entirely. Harmony requires an audio-native
understanding of genre. We achieve this via a custom CNN-Transformer hybrid architecture
(CT-GateNet) trained directly on raw audio [32][33]. The CT-GateNet architecture is specifically
chosen for its ability to capture both highly localized spectral textures (e.g., the specific transient

tension loops characteristic of Progressive House) [33]. The CNN backbone extracts local spectral
patterns, while the Transformer encoder models the global temporal evolution of the track [33].

Hierarchical Cross-Entropy Loss
Standard cross-entropy loss treats all misclassifications equally. If the model predicts Death Metal
for a Deep House track, the penalty is the same as predicting Tech House. This destroys the
model's ability to learn sonic relationships [35]. We implement a Hierarchical Cross-Entropy Loss

a consistency constraint: the probability of a child node (e.g., Afro-Organic Deep House) can never
mathematically exceed the probability of its parent node (Organic Deep House).

3.5 The Genre Graph for Discovery
The taxonomy tree built in Section 3.1 is structurally perfect for database organization, but it is
fundamentally flawed for music discovery. A tree enforces strict vertical silos. In a tree, Tipper-style
Glitch Bass (under Experimental Bass) is miles away from Glitch (under IDM). But sonically, they
share massive amounts of DNA. Similarly, Organic Deep House shares rhythmic and textural
characteristics with Melodic Techno, despite belonging to different L2 branches. The taxonomy is a
tree, but musical similarity is a graph. To build a recommendation engine that feels like magic, we
must map the horizontal, cross-genre relationships that the tree cannot capture [40][28].
Genre Node Embeddings: Every genre node is projected into a 128-dimensional embedding space
[41][42]. These embeddings are trained jointly via two signals: (1) Sonic Similarity — extracted
from the CT-GateNet classifier's penultimate layer; genres with similar spectral and temporal
features are pulled closer together. (2) Playlist Co-occurrence — extracted from user listening
habits; if users frequently place Psychedelic Bass and IDM in the same playlists, the graph edges
between those nodes strengthen [42][43]. Embeddings are stored in PostgreSQL using the
pgvector extension with an HNSW index for lightning-fast cosine similarity searches [41].
Adjacent Genre Discovery Algorithm: When a user wants to discover new music, we traverse the
genre graph to find adjacent, unexplored territories using a 2-hop traversal algorithm with a decay
discount. 1-hop neighbors receive direct sonic/playlist overlap scores. 2-hop neighbors receive a
0.7 discount factor (score_1 times score_2 times 0.7) to prevent the algorithm from drifting too far
from the user's taste profile [40].
Anti-Filter-Bubble Guarantee: Harmony enforces a strict mathematical diversity floor at the L2
level. No single L2 genre can dominate more than 70% of recommendations. If violated, the
lowest-scoring tracks from the dominant genre are replaced with the highest-scoring tracks from
underrepresented genres. The 70% cap is a floor, not a ceiling — in practice, a well-tuned
recommendation engine should naturally produce diverse output.
SPOTIFY CANNOT DO THIS: Spotify's genre system is flat and popularity-weighted. Harmony's
hierarchical genre graph enables discovery 3–4 levels deeper than any existing platform. When a

4.1 Architecture Strengths
• Denormalized Aggregates: Storing stream_count, total_tracks, and follower_count directly on

• Interaction Weighting: Pre-computing interaction_weight in the interaction_events table is a

• Cold Start Architecture: The artist_catalog_embeddings table is a perfect implementation of

• TTL Caching: The recommendation_cache table with an expires_at column prevents the

4.2 Critical pgvector Optimization: Upgrade to HNSW
The schema currently uses ivfflat indexes for the vector columns. While ivfflat is standard, it has a
major operational flaw for a fast-moving startup: it requires the table to be populated with data
before the index is built (so it can calculate the k-means centroids). If you build an ivfflat index on
an empty table, it will be useless, and you will have to manually REINDEX it as the table grows.
Since you are using a modern PostgreSQL provider (Neon), you have access to pgvector 0.5.0+,
which includes HNSW (Hierarchical Navigable Small World) indexes. HNSW can be created on an
empty table and updates dynamically as you insert rows, offers significantly higher recall and faster
query speeds than ivfflat, and eliminates the need to tune the lists parameter based on row count.
Change your index definitions from: CREATE INDEX idx_clap_embedding_ivfflat ON
track_audio_features USING ivfflat (clap_embedding vector_cosine_ops) WITH (lists = 100); To:
CREATE INDEX idx_clap_embedding_hnsw ON track_audio_features USING hnsw

4.3 Operational Blindspots to Watch
• Partitioning interaction_events: The schema notes adding partitioning when the table hits
10,000,000 rows. Do not wait until 10,000,000 rows. Implement PARTITION BY RANGE
(created_at) by month from day 1. Retroactively partitioning a massive live table requires
heavy locking and will cause downtime. Setting it up on day 1 costs nothing.
• JSONB GIN Indexing: A standard GIN index on a JSONB column is great for checking if a
key exists, but it is not optimized for range queries on the values (e.g., finding users where
'Electronic' > 0.80). If you need to query by specific genre scores, you will need a B-tree
expression index: CREATE INDEX idx_user_genre_electronic ON user_embeddings
(((genre_affinity->>'Electronic')::numeric));
• Audio URL Storage: The audio_url in the tracks table will point to your Cloudflare R2
buckets. Ensure these URLs are stored as relative paths or use a consistent CDN domain
structure so you don't have to run massive UPDATE statements if your Cloudflare routing
changes.

Section 6: Evaluation Framework
6.1 Why Evaluation Is Non-Negotiable
Recommendation systems without rigorous evaluation are just vibes. In the modern streaming
landscape, algorithms are black boxes optimized for platform retention rather than listener
expansion. Harmony's core claim — that it discovers independent artists better than any existing
platform — must be mathematically provable. These metrics are the proof. We are not building a
random shuffle; we are building a precision instrument for sonic discovery. If we cannot measure
serendipity, we cannot engineer it.

6.2 Core Metrics
Metric 1: nDCG@10 (Normalized Discounted Cumulative Gain)
nDCG is the industry standard for evaluating ranking quality, specifically because it handles graded
relevance rather than binary pass/fail [60][61]. For music recommendations, the distinction
between a track a user played on repeat and a track they skipped after 30 seconds is critical
[62][63]. nDCG@10 evaluates the top 10 recommendations, applying a logarithmic position-based
discount so that highly relevant items placed at the very top of the list are rewarded heavily
[60][64]. Formula: DCG@10 = £(i=1 to 10) rel_i / log2(i+1); nDCG@10 = DCG@10 / IDCG@10.
Relevance: rel_i = 3 if user played >80% of track; 2 if played 30–80%; 1 if played <30% but didn't
skip immediately; 0 if skipped within 10 seconds.
Metric 2: Precision@K
Precision@K = (# relevant tracks in top K) / K. A track is 'relevant' if the user played >50% of it
within 7 days of recommendation. Target: >25% at K=10. This means at least 2–3 of every 10
recommendations result in genuine engagement. While nDCG measures ranking quality,
Precision@10 provides a highly interpretable baseline of raw hit rate [61].
Metric 3: Serendipity Score
Serendipity(track) = (1 - similarity_to_recent_history) * engagement_rate, where
similarity_to_recent_history = max cosine similarity between track's CLAP embedding and user's
last 20 played tracks; engagement_rate = 1 if user played >50%, 0 otherwise. Serendipity@10 =
mean Serendipity across top 10 recommendations. Target: >0.15. A serendipity score of 0.15
means 15% of recommendations are both genuinely surprising AND engaging — the holy grail of
discovery.
Why This Matters: Spotify optimizes for engagement, which leads to filter bubbles. Harmony
optimizes for serendipity — the feeling of discovering something you didn't know you needed. This
is the emotional core of the product.
Metric 4: Diversity Score
Diversity@10 = (1 / C(10,2)) * £(ineqj) CLAP_distance(track_i, track_j), where CLAP_distance = 1
- cosine_similarity(embedding_i, embedding_j) and C(10,2) = 45 pairs. Target: >0.4. A diversity
score of 0.4 means the average pair of recommended tracks is meaningfully different from each
other, preventing the 'wall of sound' fatigue common in algorithmic radio.
Metric 5: Independent Artist Discovery Rate (The Moat Metric)
IADR = (# recommendations from artists with < 10K total streams) / total recommendations. Target:
>60%. Compare to Spotify's estimated <5% (based on published research on major label
dominance in algorithmic recommendations). When Harmony achieves IADR >60% vs Spotify's
Built with Spine AI on 2026-06-23 30
~5%, that's a 12x improvement in independent artist discovery. That's the headline.
Metric 6: Cold Start Performance
Cold Start nDCG@10 = nDCG@10 computed only for tracks with <100 total streams. Target:
>0.25. Cold start is the hardest problem in recommendation — new tracks have no interaction
history. Harmony's CLAP-based audio similarity means even brand-new tracks can be
recommended accurately based on their sonic fingerprint alone.
Metric 7: Genre Depth Score
Genre Depth Score = mean(genre_level of recommended tracks), where genre_level = 4 if
genre_l4 is populated; 3 if genre_l3 is populated; 2 if genre_l2 is populated; 1 otherwise. Target:
>2.5. This measures whether Harmony is actually using its genre depth advantage. A score of 2.5
means the average recommendation is classified at L2.5 — between subgenre and sub-subgenre
level.

7.8 The pgvector Advantage
The decision to use pgvector inside PostgreSQL rather than a dedicated vector database (like
Pinecone, Weaviate, or Qdrant) is a massive strategic advantage for a solo engineer. At 1M tracks
with 512-dim CLAP embeddings: Storage: 1M times 512 times 4 bytes = ~2GB for embeddings
alone. Quantization: Using pgvector 0.7.0+ halfvec (16-bit floats) cuts this memory footprint in half
with negligible recall loss [77][78]. HNSW Index: ~4GB additional memory. HNSW is the production
standard for evolving datasets because it maintains >95% recall without requiring the frequent
rebuilds that IVFFlat demands [75][76][79]. Performance: ANN search at 1M vectors resolves in
~10–50ms with HNSW tuned to m=16 and ef_construction=64 [75][80]. Furthermore, pgvector
0.8.0 introduces iterative scans, which drastically improves speed and relevance for filtered
queries (e.g., 'find similar tracks within this specific genre') [78]. This entire workload fits
comfortably in Supabase's $200/mo Enterprise tier.
Infrastructure Philosophy: The best infrastructure for a solo engineer is the infrastructure that
doesn't require a dedicated DevOps engineer. Railway + Supabase + pgvector is that stack. It
scales from $11/mo to $1,350/mo without a single architecture change — just bigger instances,

better index tuning, and zero egress fees.

Harmony Discovery Engine
Deep Research & Technical Specification — The Complete Blueprint
2026
Executive Summary
Harmony is building an audio-first music recommendation engine that solves the most fundamental
problem in music discovery: the cold start problem. Every major streaming platform — Spotify,
Apple Music, YouTube Music, SoundCloud — requires a track to accumulate thousands of listener
interactions before its recommendation algorithm can meaningfully surface it. Harmony eliminates

1.2 Apple Music — The Editorial Gatekeeping Problem
Apple Music represents a fundamentally different philosophical approach to music
recommendation. Rather than relying purely on deep learning and interaction graphs, Apple
employs a hybrid system that heavily weights human editorial curation alongside algorithmic
amplification. The top of the funnel is controlled by human editors who curate flagship playlists
(e.g., New Music Daily, Today's Hits, A-List by Genre). The algorithmic layer — powering surfaces
like the New Music Mix, Personal Mix, and Now Playing radio — is built on top of these editorial
signals and collaborative filtering derived from the Apple ecosystem.
Apple Music's discovery algorithm in 2026 relies on seven primary signals [11]: Shazam Tag
Volume (~2.1 million tags per day in 2024) [11]; Library Add Rate as a strong signal indicating high
user intent [11]; Replay Completion Rate measuring if a user finishes a track after starting it [11];
the 'Discovered Listeners' Metric tracking listeners who heard the artist through editorial or
algorithmic surfaces [11]; Now Playing Auto-Radio Session Length [11]; Geographic Listening
Patterns for localized popularity spikes [11]; and Spatial Audio (Dolby Atmos) Engagement, which
contributes roughly a 10% payout lift and heavily influences algorithmic recommendations [11].
Apple Music's fatal flaw for independent artists is its editorial gatekeeping. Editorial pitches are a
one-time human evaluation [11]. If an independent artist is not 'picked up' by Apple's editorial team,
they are effectively invisible to the broader ecosystem. The algorithmic surfaces compound from
early listener signals — specifically first-week saves and replays (roughly the first 7–14 days
post-release) [11]. However, generating those initial signals requires an audience. The algorithm
acts as an amplifier for editorial choices and established artists. Undiscovered artists without a
pre-existing fanbase or editorial backing cannot generate the initial velocity required to trigger the
algorithmic compounding. There is no robust audio-first cold start mechanism. [11]
THE HARMONY ADVANTAGE: Harmony is fully algorithmic from day one. There is no editorial

1.5 The Five Universal Failure Patterns
When analyzing Spotify, Apple Music, YouTube Music, and SoundCloud at the architectural level,
five distinct failure modes emerge. These are not bugs; they are the intended mathematical
outcomes of their system designs.
1. Popularity Bias (Gradient Starvation): All four platforms use interaction volume as their
primary training signal proxy. Whether it is Spotify's co-listening HGNN [3], YouTube's
watch-time weighted logistic regression [15], or Apple's reliance on Shazam volume [11], the
models are mathematically biased toward tracks with massive historical data. The rich get
richer by design.
2. Cold Start Blindness: None of the incumbents possess a robust, purely audio-first cold start
mechanism that scales. Spotify requires session co-occurrence [7], Apple requires editorial
selection or early compounding [11], and SoundCloud's 'First Fans' is a limited 100-user
band-aid over a social graph [20].
3. Metric Misalignment: YouTube and Spotify optimize for engagement, session length, and
retention [1][16]. This metric misalignment prioritizes passive, familiar background listening
over active, novel discovery, actively suppressing niche genres that require attentive listening.
4. Metadata Dependency: Platforms rely heavily on NLP of lyrics, blog sentiment, and
user-supplied tags [7][1]. For underground electronic, experimental bass, or instrumental
niche genres, this metadata is either non-existent or highly inaccurate, rendering text-based
retrieval useless.
5. Business Model Conflicts: The most critical failure point. Spotify's Discovery Mode, Apple's
major label relationships, and YouTube's ad-inventory requirements mean that even if their
engineers wanted to flatten the popularity curve to favor independent artists, their executive
boards would veto it. The bias is monetized.

2.2 Layer A: Audio Similarity Engine (CLAP + Essentia)
CLAP (Contrastive Language-Audio Pretraining) produces a holistic 512-dimensional semantic
embedding — it understands that a track 'sounds like late-night driving music' or 'has the energy of
a festival closer.' But CLAP is a black box. You can't interrogate why two tracks are similar.
Essentia gives you the forensics: exact BPM, MFCC coefficients, spectral centroid, key/scale.
These features are interpretable, filterable, and composable. You can say 'find me tracks within
plusminus5 BPM of 128, in a minor key, with high danceability.' Together: CLAP finds the semantic
neighborhood. Essentia filters and re-ranks within it. The result is recommendations that feel both
intuitively right (CLAP) and musically coherent (Essentia).

2.3 Layer B: Collaborative Filtering (ALS with Popularity Penalty)
Collaborative filtering answers the question: 'What do people with similar taste to this user love?' It
doesn't care about audio at all — it's pure behavioral signal. A user who loves Burial, Andy Stott,
and Actress will get recommended Demdike Stare not because it sounds similar (though it does)
but because the same humans who love those artists also love Demdike Stare. For independent
artists, this is both the most powerful signal and the most dangerous one. Powerful because it
captures taste in ways audio analysis never can. Dangerous because it naturally amplifies popular
tracks — the rich get richer. The solution: a popularity penalty that inverts the rich-get-richer
dynamic.

captures. A track can be sonically similar to another but in a completely different genre context. A
user can love jazz and also love electronic music — two genres that rarely appear in the same
collaborative filtering neighborhood. The genre graph makes these structural relationships explicit
and learnable.

Section 3: The Niche Genre Discovery System
3.1 The Genre Taxonomy
The fundamental failure of modern music recommendation engines is their shallow understanding
of genre. Spotify, Apple Music, and Tidal operate almost exclusively at Level 1 (L1) or Level 2 (L2)
genre specificity. To them, a track is 'Electronic' (L1) or 'House' (L2). For a casual listener, this is
sufficient. But for the obsessive music fan — the core demographic Harmony is built for — this
shallow categorization is useless. A listener searching for the organic, polyrhythmic textures of
'Afro-Organic Deep House' (L4) will be actively alienated if an algorithm serves them 'Big Room
Progressive' (L3) simply because both roll up to 'House' [27]. Operating at L4/L5 genre specificity
is Harmony's core competitive advantage. No platform has ever built a recommendation engine at
this depth because the data engineering required to maintain a 6,000+ node taxonomy and the
machine learning required to classify audio into it are exceptionally difficult.
Data Sources for Taxonomy Construction
• Every.Noise.at: The holy grail of genre data — a publicly accessible frontend for Spotify's
internal genre map containing over 6,000+ genres [27]. The scraping strategy: parse the
DOM, extract text nodes of the links to form the raw taxonomy, and map the associated
playlist URIs to extract reference tracks for training [29][30].
• MusicBrainz: An open-source, community-maintained database offering over 1,000+ genres
[27]. Access via their open API by paginating through
https://musicbrainz.org/ws/2/genre?limit=100&offset=0&fmt=json [31].
• Discogs: The definitive database for electronic and physical releases, containing over 15M+
releases [28]. Uses a two-tier system: Genre is top-level, Style is the subgenre. Extract via
monthly XML data dumps, mapping Style tags to L3/L4 nodes [28].
• Rate Your Music (RYM): Possesses the most pedantic, highly debated community genre
taxonomy on the internet. Scrape their genre hierarchy pages to understand parent-child
relationships between obscure micro-genres, critical for structuring L4 and L5 nodes.

3.2 The CT-GateNet CNN-Transformer Genre Classifier
To operate at L4/L5 specificity, metadata alone is insufficient. User-generated tags are notoriously
noisy, and emerging artists often lack metadata entirely. Harmony requires an audio-native
understanding of genre. We achieve this via a custom CNN-Transformer hybrid architecture
(CT-GateNet) trained directly on raw audio [32][33]. The CT-GateNet architecture is specifically
chosen for its ability to capture both highly localized spectral textures (e.g., the specific transient
snap of a Neurofunk snare) and long-term structural dependencies (e.g., the 16-bar unresolved
tension loops characteristic of Progressive House) [33]. The CNN backbone extracts local spectral
patterns, while the Transformer encoder models the global temporal evolution of the track [33].

3.3 The Genre Graph for Discovery
The taxonomy tree built in Section 3.1 is structurally perfect for database organization, but it is
fundamentally flawed for music discovery. A tree enforces strict vertical silos. In a tree, Tipper-style
Glitch Bass (under Experimental Bass) is miles away from Glitch (under IDM). But sonically, they
share massive amounts of DNA. Similarly, Organic Deep House shares rhythmic and textural
characteristics with Melodic Techno, despite belonging to different L2 branches. The taxonomy is a
tree, but musical similarity is a graph. To build a recommendation engine that feels like magic, we
must map the horizontal, cross-genre relationships that the tree cannot capture [40][28].

Section 4: Complete PostgreSQL Database Schema
4.1 Schema Architecture Overview
This PostgreSQL schema is architected for a solo engineer building a high-scale recommendation
engine. It correctly balances normalized relational data with denormalized JSONB/aggregate
columns to prevent expensive JOIN and COUNT() operations at runtime. The integration of
pgvector alongside standard relational data ensures the ALS and CLAP embeddings can be
queried in the same transaction as metadata filtering.
• Denormalized Aggregates: Storing stream_count, total_tracks, and follower_count directly on
the artists and tracks tables is the correct move. Calculating these on the fly via COUNT()
would cripple the database at 100,000 users.
• Interaction Weighting: Pre-computing interaction_weight in the interaction_events table is a
massive time-saver for the Python ALS training job. It shifts the compute burden from the
batch job to the event ingestion phase.
• Cold Start Architecture: The artist_catalog_embeddings table is a perfect implementation of
the ACARec approach. By pre-aggregating the CLAP and CF embeddings at the artist level,
you guarantee that a brand-new track from an existing artist instantly inherits a valid
recommendation vector.

• TTL Caching: The recommendation_cache table with an expires_at column prevents the

4.2 Critical pgvector Optimization: HNSW over IVFFlat
IVFFlat has a major operational flaw for a fast-moving startup: it requires the table to be populated
with data before the index is built (so it can calculate the k-means centroids). If you build an
IVFFlat index on an empty table, it will be useless, and you will have to manually REINDEX it as
the table grows. HNSW (Hierarchical Navigable Small World) indexes, available in pgvector 0.5.0+,
can be created on an empty table and update dynamically as you insert rows. They offer
significantly higher recall and faster query speeds than IVFFlat, and eliminate the need to tune the
lists parameter based on row count.

6.1 Why Evaluation Is Non-Negotiable
Recommendation systems without rigorous evaluation are just vibes. In the modern streaming
landscape, algorithms are black boxes optimized for platform retention rather than listener
expansion. Harmony's core claim — that it discovers independent artists better than any existing
platform — must be mathematically provable. These metrics are the proof. We are not building a
random shuffle; we are building a precision instrument for sonic discovery. If we cannot measure
serendipity, we cannot engineer it.
6.2 Core Metrics
Metric 1: nDCG@10 (Normalized Discounted Cumulative Gain)
nDCG is the industry standard for evaluating ranking quality, specifically because it handles graded
relevance rather than binary pass/fail [60][61]. For music recommendations, the distinction
between a track a user played on repeat and a track they skipped after 30 seconds is critical
[62][63]. nDCG@10 evaluates the top 10 recommendations, applying a logarithmic position-based
discount so that highly relevant items placed at the very top of the list are rewarded heavily [60][64].
DCG@10 = £(i=1 to 10) rel_i / log2(i+1)nDCG@10 = DCG@10 / IDCG@10where rel_i = relevance
of track at position i= 3 if user played >80% of track= 2 if user played 30-80%= 1 if user played
<30% but didn't skip immediately= 0 if skipped within 10 seconds
Metric 2: Precision@K
Precision@K = (# relevant tracks in top K) / KA track is 'relevant' if the user played >50% of it
within 7 days of recommendation.Target: > 25% at K=10. This means at least 2-3 of every 10
recommendations result in genuine engagement.
Metric 3: Serendipity Score
Serendipity(track) = (1 - similarity_to_recent_history) * engagement_ratewhere:
similarity_to_recent_history = max cosine similarity between track's CLAP embeddingand user's
last 20 played tracksengagement_rate = 1 if user played >50%, 0 otherwiseSerendipity@10 =
mean Serendipity across top 10 recommendationsTarget: > 0.15. A serendipity score of 0.15
means 15% of recommendations are both genuinely surprising AND engaging — the holy grail of
discovery.
Metric 4: Diversity Score
Diversity@10 = (1 / C(10,2)) * £(ineqj) CLAP_distance(track_i, track_j)where CLAP_distance = 1 -
cosine_similarity(embedding_i, embedding_j)C(10,2) = 45 pairsTarget: > 0.4. A diversity score of
0.4 means the average pair of recommended tracks is meaningfully different from each other,
preventing 'wall of sound' fatigue common in algorithmic radio.

Metric 5: Independent Artist Discovery Rate (The Moat Metric)
IADR = (# recommendations from artists with < 10K total streams) / total recommendationsTarget:
> 60%. Compare to Spotify's estimated < 5% (based on published research on major label
dominance in algorithmic recommendations). When Harmony achieves IADR > 60% vs Spotify's
~5%, that's a 12x improvement in independent artist discovery. That's the headline.
Metric 6: Cold Start Performance
Cold Start nDCG@10 = nDCG@10 computed only for tracks with < 100 total streamsTarget: >
0.25. Cold start is the hardest problem in recommendation — new tracks have no interaction
history. Harmony's CLAP-based audio similarity means even brand-new tracks can be
recommended accurately based on their sonic fingerprint alone.
Metric 7: Genre Depth Score
Genre Depth Score = mean(genre_level of recommended tracks)where genre_level = 4 if genre_l4
is populated= 3 if genre_l3 is populated= 2 if genre_l2 is populated= 1 otherwiseTarget: > 2.5. This
measures whether Harmony is actually using its genre depth advantage.

7.8 The pgvector Advantage
The decision to use pgvector inside PostgreSQL rather than a dedicated vector database (like
Built with Spine AI on 2026-06-23 43
Pinecone, Weaviate, or Qdrant) is a massive strategic advantage for a solo engineer. At 1M tracks with 512-dim CLAP embeddings:
• Storage: 1M times 512 times 4 bytes = ~2GB for embeddings alone.
• Quantization: Using pgvector 0.7.0+ halfvec (16-bit floats) cuts this memory footprint in half
with negligible recall loss [77][78].
• HNSW Index: ~4GB additional memory. HNSW is the production standard for evolving
datasets because it maintains >95% recall without requiring the frequent rebuilds that IVFFlat
demands [75][76][79].
• Performance: ANN search at 1M vectors resolves in ~10-50ms with HNSW tuned to m=16
and ef_construction=64 [75][80]. pgvector 0.8.0 introduces iterative scans, which drastically
improves speed and relevance for filtered queries (e.g., 'find similar tracks within this specific
genre') [78].
• This entire workload fits comfortably in Supabase's $200/mo Enterprise tier.
