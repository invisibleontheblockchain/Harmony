Section 7: Critical Decisions Needed Before Building
Decision 1: Monorepo Package Manager
Table 24 — Decision 1: Monorepo Package Manager
Built with Spine AI on 2026-06-23 59
Field Detail
Context The repo uses npm workspaces but has no
per-package package.json files. The monorepo is
not fully configured. This affects every build
command, every CI step, and every deployment
pipeline.
Option A npm workspaces (current) — minimal config,
familiar, but slow installs and permissive hoisting
allows phantom dependency bugs.
Option B pnpm workspaces — faster installs, strict hoisting
prevents phantom dependencies, better for
monorepos.
Option C Turborepo + pnpm — best for monorepo at scale.
Turborepo adds build caching (saves significant CI
time), task orchestration, and remote caching.
pnpm provides strict dependency isolation.
Recommendation Option C — Turborepo + pnpm. The build pipeline
complexity (TypeScript API + Next.js web + Expo
mobile + Python microservice + Cloudflare Worker)
justifies the setup cost. Remote caching will save
hours of CI time per week.
Deadline Sprint 1, Day 1 — before any new packages are
added. Changing package managers after the fact
is painful.
Decision 2: API Architecture
Table 25 — Decision 2: API Architecture
Field Detail
Context The repo has a Fastify API at apps/api. The
blueprint mentions Next.js API routes. These are
two different architectures. Mixing them creates
confusion about where business logic lives.
Option A Keep Fastify as standalone API (current) —
already built, handles WebSockets natively, better
for royalty worker and stream processing, no
framework lock-in.
Built with Spine AI on 2026-06-23 60
Option B Migrate to Next.js API routes — collocates web
frontend and API, but loses WebSocket support,
worse for background jobs, and requires rewriting
all existing routes.
Option C Keep Fastify + add Next.js for web-only routes —
hybrid approach, but creates confusion about
which layer handles what.
Recommendation Option A — Keep Fastify. It is already built and
tested. It handles WebSockets natively (required
for real-time messages). It is better suited for the
royalty worker and stream processing. Next.js API
routes are fine for simple CRUD but Fastify is the
right tool for this use case.
Deadline Sprint 1 — must be decided before building the
web app so the API client is configured correctly.
